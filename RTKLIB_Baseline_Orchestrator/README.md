## Overview

RTKLIB Baseline Orchestrator is a Python-based modular workflow for reproducible RTKLIB/rnx2rtkp baseline processing.

The program was developed to support short-baseline static processing between a CORS/base station and one or more GNSSBM rover occupations near a tide gauge. The CORS/base station coordinates are assumed to have been estimated independently, for example by the GINAN/pea PPP Batch Processor. The workflow reads rover and base RINEX observation files, extracts RINEX metadata and observation time spans, matches rover and base files by time overlap, resolves GNSS navigation and precise products, builds deterministic RTKLIB configuration files, runs `rnx2rtkp`, parses POS outputs, performs fixed/float and final-window QC, estimates GNSSBM coordinates and generates an auditable HTML processing report.

The current implementation supports both command-line execution and a Tkinter-based graphical interface.

## Main Capabilities

The current MVP implementation supports the following processing chain:

1. System preflight checks.
2. Parsing of a CORS GINAN PPP HTML report for the reference station coordinate solution.
3. Discovery of rover GNSSBM RINEX observation files.
4. Discovery of CORS/base RINEX observation files.
5. RINEX header parsing.
6. Extraction of station metadata, receiver/antenna metadata, approximate coordinates and observation time span.
7. Rover–base time-overlap matching.
8. Identification and validation of required broadcast and/or precise GNSS products.
9. RTKLIB 2.4.2 p13 compatibility preprocessing for modern MGEX SP3 products when required.
10. Deterministic RTKLIB configuration generation for each accepted rover–base pair.
11. Execution of RTKLIB `rnx2rtkp`.
12. Parsing of RTKLIB `.pos` outputs in ECEF XYZ format.
13. Fixed/float solution quality-control summary.
14. Extraction of one final coordinate solution per rover occupation from the selected final time window.
15. Computation of rover ECEF, geodetic and local baseline coordinates.
16. Generation of `final_baseline_solutions.csv`.
17. Generation of an HTML processing report, `bases.solution.report.html`.
18. Warning-aware reporting for low fixed-percentage runs and other QC flags.

## Processing Model

The processor follows a per-occupation short-baseline processing strategy.

Each accepted rover/base RINEX pair is treated as one independent processing unit. For each pair, the software generates a dedicated RTKLIB configuration file and runs `rnx2rtkp` independently. The resulting POS time series is then reduced to one representative GNSSBM coordinate solution using the final-window settings and QC criteria selected by the user.

The processing model is:

```text
CORS GINAN PPP report
-> rover/base RINEX discovery
-> RINEX header time-span extraction
-> rover-base overlap matching
-> product/model resolution
-> optional SP3 compatibility preprocessing
-> deterministic RTKLIB configuration generation
-> rnx2rtkp execution
-> POS parsing and QC
-> per-run GNSSBM coordinate solution extraction
-> final_baseline_solutions.csv
-> bases.solution.report.html
```

## User Inputs

The GUI and command-line workflow request the following main inputs.

### CORS GINAN PPP report

Path to the HTML report generated by the GINAN/pea PPP Batch Processor for the CORS/base station.

The RTKLIB Baseline Orchestrator extracts the reference station solution from this report, including the CORS identifier and geocentric coordinates:

```text
X_m
Y_m
Z_m
```

If available in the parsed GINAN output, the CORS report may also contain geodetic and local coordinate fields such as:

```text
lon_deg
lat_deg
h_m
E_m
N_m
U_m
```

The CORS ECEF coordinate solution is used as the reference position in the RTKLIB baseline processing.

### Rover GNSSBM RINEX root directory

Root directory containing the GNSS benchmark rover RINEX observation files.

Each rover RINEX file is treated as a candidate rover occupation. For example:

```text
RAW_rinex/
├── R1__0790.26o
├── R3__0790.26o
└── RTK_process/
```

Generated outputs are written under `RTK_process/` inside the rover RINEX root, unless the user explicitly selects another output folder.

### CORS/base RINEX root directory

Root directory containing the CORS/base RINEX observation files.

The workflow discovers candidate base files and matches them against rover files using observation time overlap. The matching strategy can be configured by the user.

### Products / models directory

Directory containing broadcast navigation files, precise orbit files, precise clock files and optional auxiliary models.

Typical contents may include:

```text
BRDC00IGS_R_<YYYYDDD>0000_01D_MN.rnx
COD0MGXFIN_<YYYYDDD>0000_01D_05M_ORB.SP3
COD0MGXFIN_<YYYYDDD>0000_01D_30S_CLK.CLK
igs20.atx
```

### RTKLIB executable path

Path to the `rnx2rtkp` executable used for processing.

The reference setup uses RTKLIB 2.4.2 p13 compiled natively under Linux, for example:

```text
/home/ioannis/GNSS_TEST_WORK/rtklib_native/RTKLIB_2.4.2_p13/app/rnx2rtkp/gcc/rnx2rtkp
```

### Product provider, series and project

The user selects the GNSS product family to be used, for example:

```text
Provider: COD
Series:   FIN
Project:  MGX
```

These selections are product-family selectors for product resolution and compatibility preprocessing. They are not PPP-only options.

### Processing mode

The main validated processing mode is:

```text
static
```

A dynamic mode is present as a GUI/CLI option, but it requires additional validation before operational use.

### Rover/base matching strategy

The supported matching strategies are:

```text
best_overlap_per_rover
all_valid_overlaps
```

- `best_overlap_per_rover`: keeps the best base overlap for each rover file.
- `all_valid_overlaps`: keeps all rover/base overlaps satisfying the minimum overlap threshold.

### Minimum overlap

Minimum required rover/base overlap in minutes.

Pairs with shorter overlap are rejected before RTKLIB processing.

### RTKLIB processing options

The workflow exposes the main RTKLIB processing options used by `rnx2rtkp`, including:

```text
processing_mode
frequency_mode
solution_type
elevation_mask_deg
ambiguity_mode
ambiguity_threshold
nav_systems
output_coordinate_format
trace_level
```

For the current static baseline workflow, the primary output coordinate format is ECEF XYZ.

### Final-window solution settings

The final coordinate solution is computed from the last selected time window of the RTKLIB POS solution.

Current settings include:

```text
final_window_minutes
q_fixed_only_for_final
min_fixed_percent
min_ratio_for_fixed
```

By default, the final coordinate solution is computed from Q=1 fixed epochs only within the final window.

### Execution mode

Two execution modes are supported:

```text
run
build_only
```

- `run`: builds RTKLIB configuration files and executes `rnx2rtkp`.
- `build_only`: builds configuration and command files but skips `rnx2rtkp` execution.

### Overwrite option

Controls whether existing generated run outputs may be overwritten.

### Report generation

If enabled, the processor builds:

```text
bases.solution.report.html
```

The report is written under the selected output folder.

### Core Python modules

- `paths_config.py`  
  Defines default paths, product options, processing options, matching strategies and validation constants.

- `models.py`  
  Defines the main dataclasses used by the workflow, including user inputs, RINEX metadata, resolved products, run configurations, run results, CORS solutions and baseline solutions.

- `system_check.py`  
  Performs preflight checks for executables, required input paths, output directories and optional downloader resources.

- `cors_report_parser.py`  
  Parses the CORS GINAN PPP HTML report and extracts the reference station coordinate solution.

- `rinex_header.py`  
  Discovers RINEX observation files, parses RINEX headers and extracts station metadata, receiver/antenna metadata, approximate coordinates and observation time spans.

- `overlap_matcher.py`  
  Matches rover and base RINEX files by observation time overlap and applies the selected matching strategy.

- `products_resolver.py`  
  Resolves broadcast navigation, precise orbit, precise clock, ANTEX and optional auxiliary products for each rover/base processing pair.

- `sp3_compat.py`  
  Converts modern MGEX SP3-d products into RTKLIB 2.4.2-compatible SP3-c-like subsets when required, retaining only the selected GNSS systems and avoiding the RTKLIB 2.4.2 `>99 satellites` limitation.

- `rtklib_conf_builder.py`  
  Builds deterministic RTKLIB configuration files and complete `rnx2rtkp` command lines for each accepted rover/base pair.

- `rnx2rtkp_runner.py`  
  Executes `rnx2rtkp`, captures stdout/stderr and records run status.

- `pos_parser.py`  
  Parses RTKLIB `.pos` files in ECEF XYZ output format and returns structured POS time-series data.

- `baseline_qc.py`  
  Computes POS QC statistics, final-window coordinate solutions, baseline ENU components, geodetic coordinates and local scatter metrics.

- `report_builder.py`  
  Generates the HTML `bases.solution.report.html`, including processing metadata, inventories, product resolution, execution summary, per-run coordinate solutions, QC flags and plots.

- `cleanup_service.py`  
  Provides cleanup utilities for generated run folders, logs, tables and reports.

- `batch_main.py`  
  Top-level orchestrator that connects all modules into a complete batch-processing workflow and provides CLI execution.

- `rtklib_baseline_gui.py`  
  Tkinter graphical user interface for launching RTKLIB baseline processing.

## SP3 Compatibility Logic

RTKLIB 2.4.2 p13 does not reliably process modern MGEX SP3-d files containing more than 99 satellites.

For this reason, `sp3_compat.py` can generate a reduced SP3-c-like compatible product for RTKLIB 2.4.2. The compatibility product retains only the selected GNSS systems required by the run, for example:

```text
G,E
```

and writes a compatible file under:

```text
RTK_process/compatible_products/
```

The original product is not modified.

## Coordinate Convention

### ECEF coordinates

The primary RTKLIB POS parsing and final coordinate extraction are based on ECEF XYZ coordinates:

```text
X_m
Y_m
Z_m
```

The ECEF coordinates are read from the RTKLIB POS output generated with:

```text
out-solformat = xyz
```

### WGS84 geodetic coordinates

Rover geodetic coordinates are derived from the final ECEF solution using the same convention and numerical logic as RTKLIB 2.4.2 p13 `ecef2pos()`.

The constants are:

```text
RE_WGS84 = 6378137.0
FE_WGS84 = 1.0 / 298.257223563
```

The transformation is:

```text
X_m, Y_m, Z_m -> lat_rad, lon_rad, h_m
```

and the reported angular coordinates are converted to degrees:

```text
lon_deg
lat_deg
h_m
```

The reported height `h_m` is WGS84 ellipsoidal height.

### Local baseline ENU coordinates

The baseline vector is first computed in ECEF:

```text
dX = X_rover - X_cors
dY = Y_rover - Y_cors
dZ = Z_rover - Z_cors
```

It is then transformed to local ENU at the CORS reference position:

```text
baseline_E_m
baseline_N_m
baseline_U_m
```

where:

- `baseline_E_m` is the east component of the rover relative to the CORS.
- `baseline_N_m` is the north component of the rover relative to the CORS.
- `baseline_U_m` is the up component of the rover relative to the CORS.

### Local scatter fields

The fields:

```text
std_lon_m
std_lat_m
std_h_m
```

are not angular standard deviations in degrees.

They are local metric scatter values computed from the final-window epoch residuals around the final rover solution:

```text
X_i - X_mean
Y_i - Y_mean
Z_i - Z_mean
```

These residuals are transformed to local ENU at the final rover coordinate. The reported fields therefore mean:

- `std_lon_m`: standard deviation of the local east component, approximately the metric scatter along longitude.
- `std_lat_m`: standard deviation of the local north component, approximately the metric scatter along latitude.
- `std_h_m`: standard deviation of the local up component, corresponding to ellipsoidal-height scatter.

They are intended as final-window internal precision indicators, not as full external coordinate uncertainty estimates.

The report includes:

6. Rover and CORS/base RINEX inventories.
7. Rover–base overlap matching table.
8. Product and model resolution table.
9. RTKLIB processing options.
10. Execution summary.

13. Final GNSSBM coordinate solution note.
14. Warnings and rejected runs.

The per-run baseline solution table includes both geocentric and geodetic rover coordinates:

```text
X_m
Y_m

```

and both ECEF and local metric scatter indicators:

```text
std_X_m
std_Y_m
std_Z_m
std_lon_m
std_lat_m
std_h_m
```

For static mode, the report plot section focuses on baseline geometry, CORS position, solved rover point(s), horizontal error ellipse proxy and height accuracy bar.

For dynamic mode, the report plot section focuses on rover trajectory points. Dynamic mode remains less validated than static mode in the current MVP.

### Satellite basemap dependency

Generation of satellite-image maps in `bases.solution.report.html` requires the Python package:

```text
contextily
```

It must be installed in the Python environment that runs the RTKLIB Baseline Orchestrator, GUI or notebook. For example:

```bash
python -m pip install contextily
```

If `contextily` is not available, the report can still be generated, but the Section 15 map will not include the satellite-image basemap.

When installing `contextily` inside an existing Anaconda environment, the user should check that the installation does not upgrade core packages such as `numpy` to versions incompatible with other installed packages.


## Failure Handling

The processor distinguishes between run statuses such as:

```text
SUCCESS
SKIPPED_EXISTING
FAILED
```

If a run succeeds, its POS file is parsed and a baseline solution is extracted.

If a run is skipped because an existing output is present and overwrite is disabled, the status is recorded in `run_index.csv`.

If a run fails, the failure is recorded and included in the report. Successful runs can still contribute to the final solution table and report.

QC flags are attached to baseline solutions when relevant. For example:

```text
LOW_FIXED_PERCENT
EMPTY_POS
NO_EPOCHS_IN_FINAL_WINDOW
FINAL_WINDOW_BELOW_RECOMMENDED_MINIMUM
```

## Optional Product Downloader Environment

The workflow can use an optional external product downloader for missing GNSS products.

In the reference setup, the downloader resources are:

```text
Downloader script = /home/ioannis/data/RINEX/ginan_batch_PPP/auto_download_PPP.py
Downloader Python = /home/ioannis/data/RINEX/ginan_batch_PPP/ginanenv/bin/python
```

These paths are system-specific and must be reviewed when the workflow is moved to another computer.

The default downloader Python path assumes that a dedicated Python virtual environment named `ginanenv` already exists under:

```text
/home/ioannis/data/RINEX/ginan_batch_PPP/
```

A similar environment can be created once on the target system, for example:

```bash
cd /home/ioannis/data/RINEX/ginan_batch_PPP
python3 -m venv ginanenv
source ginanenv/bin/activate
pip install --upgrade pip
pip install click numpy requests Hatanaka gnssanalysis matplotlib
```

The environment is used by the downloader script only. The RTKLIB Baseline Orchestrator itself may run from a different Python environment, such as Anaconda.

If automatic product downloading is not required, the user may set:

```text
Download missing products = n
```

In that case, all required broadcast/precise products must already exist in the selected products directory.

## Limitations

The current MVP remains a controlled research-processing tool.

Known limitations:

- It assumes a valid RTKLIB/rnx2rtkp installation.
- Static short-baseline mode is the primary validated workflow.
- Dynamic processing mode is present but requires further validation.
- It currently processes accepted rover/base pairs sequentially.
- The final coordinate solution is extracted per run; benchmark-level weighted combination is a later extension.
- The local scatter fields are internal final-window precision indicators, not full external uncertainty estimates.
- Product downloading is optional and depends on the configured downloader resources.
- The SP3 compatibility preprocessing is a practical solution for RTKLIB 2.4.2 p13 compatibility with modern MGEX products; the original product is preserved.

## Intended Use

The primary intended use is reproducible RTKLIB short-baseline processing of GNSS benchmark observations.

The workflow is designed to support:

- controlled processing of GNSSBM rover occupations,
- repeatable RTKLIB configuration generation,
- traceable CORS reference coordinate usage,
- traceable product selection,
- fixed/float RTKLIB solution validation,
- extraction of GNSSBM coordinate solutions,
- technical reporting for GNSS benchmark documentation.

## Notes for Porting to Another System

If the workflow is moved to another system, review and update:

- `paths_config.py`,
- RTKLIB `rnx2rtkp` executable path,
- default CORS GINAN PPP report path,
- default rover RINEX root,
- default CORS/base RINEX root,
- default products/models root,
- optional downloader paths,
- desktop launcher paths, if used.

The GUI allows the user to override the main processing paths at runtime, but the default values are still defined in the project configuration and GUI source.
